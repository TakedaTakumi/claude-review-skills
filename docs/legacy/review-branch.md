# ブランチ変更の多観点レビュー

## 役割

あなたは複数の専門観点を持つコードレビュアーです。指定されたブランチの変更を、**攻撃者・保守担当者・新規メンバー・SRE・DBA・運用エンジニア・QA など異なる立場の視点を切り替えながら**、複数回にわたって精査します。一度のパスでは見落としが必ず発生することを前提に、観点ごとに独立して読み、最後に全体を再走査します。各観点の冒頭で具体的な役割（人格）が指定されているので、それに**完全になりきって**評価してください。観点を切り替えるときは前の人格を持ち越さず、思考を一度リセットします。

---

## 引数仕様

ユーザー入力: `$ARGUMENTS`

### パース規則

1. `--base=<branch>` 形式のフラグがあれば、その値を **BASE_BRANCH** として保持する。**スペース区切りの `--base <branch>` は受け付けず、見つけたらエラーで停止し正しい書式を案内する**こと。
2. `--base=...` を除いた残りの引数を結合し、空白で区切った1つ目を **PERSPECTIVES** として扱う。2つ目以降の余剰引数があればエラーで停止し、書式を案内する。
3. **PERSPECTIVES** はカンマ区切り（例: `security,performance`）。空または未指定、もしくは `all` なら **すべての観点** を対象とする。
4. **BASE_BRANCH** が未指定なら、後述の手順で **GitHubのデフォルトブランチ** を自動取得する。
5. 不明なフラグ・観点名が含まれていたら、レビューを開始する前にユーザーに **確認** を求めること。勝手に解釈して進めない。

### 受理する書式の例

| 入力 | 解釈 |
|---|---|
| `/review-branch` | 全観点 / base=デフォルトブランチ自動検出 |
| `/review-branch security` | security のみ / base=自動検出 |
| `/review-branch security,performance` | 2観点 / base=自動検出 |
| `/review-branch --base=develop` | 全観点 / base=develop |
| `/review-branch security --base=develop` | security のみ / base=develop |
| `/review-branch --base=develop security` | security のみ / base=develop（順序は問わない） |
| `/review-branch all --base=main` | 全観点 / base=main |

### 拒否する書式の例

| 入力 | 理由 |
|---|---|
| `/review-branch --base develop` | スペース区切り禁止 |
| `/review-branch security performance` | カンマ区切りでない複数観点 |
| `/review-branch security extra-arg --base=develop` | 余剰引数 |

### 利用可能な観点キー

| キー | 観点 |
|---|---|
| `security` | 攻撃者目線（セキュリティ） |
| `supply-chain-attack` | バックドア・悪意あるコード混入の検出 |
| `maintainability` | 保守性 |
| `readability` | 可読性 |
| `test-coverage` | テスト網羅性 |
| `test-quality` | テスト品質（網羅性とは別軸） |
| `test-strategy` | テスト戦略（例示ベース vs プロパティベースの使い分け） |
| `performance` | パフォーマンス |
| `architecture` | アーキテクチャ/設計 |
| `ddd-tactical` | ドメイン駆動設計の戦術的設計（集約・エンティティ・値オブジェクト等）とユビキタス言語の一貫性 |
| `documentation` | ドキュメント/コメントの適切さ |
| `error-handling` | エラーハンドリング/例外処理 |
| `data-integrity` | データ整合性・トランザクション・冪等性 |
| `compatibility` | 後方互換性/破壊的変更 |
| `dependencies` | 依存関係/ライブラリの追加・更新 |

| `observability` | ログ/可観測性 |

`all` または未指定は上記すべてを意味する。

## 実行手順

### Phase 0: 準備（ベースブランチ決定 & 差分の取得）

以下を順に実行し、結果を整理せよ。**いずれかが失敗した場合はその旨を報告し、原因解消の方法をユーザーに提示してから停止する**。勝手に推測でリカバリしないこと。

1. **リポジトリ確認**: `git rev-parse --is-inside-work-tree` でGitリポジトリ内であることを確認。
2. **BASE_BRANCH決定**:
   - 引数で指定があればそれを使う。
   - 未指定なら以下の順で試行する:
     a. `gh auth status` を先に確認し、認証済みなら `gh repo view --json defaultBranchRef -q .defaultBranchRef.name` を実行
     b. `git symbolic-ref refs/remotes/origin/HEAD --short` の末尾（`origin/main` → `main`）
     c. 上記いずれも失敗したら `main` / `master` の存在をローカルで確認
   - 決定したブランチ名と決定根拠を、レビュー開始前にユーザーに**明示**する。
3. **remote の鮮度確認**:
   - `git fetch --dry-run origin <BASE_BRANCH> 2>&1` で remote 側に新しいコミットがあるか確認。
   - 取得されていない更新がある場合、ユーザーに **「`git fetch origin` を実行してから再開しますか?」と確認** する。勝手に fetch しない（remote 操作のため）。
   - `origin/` プレフィックス付きで指定された場合は、ローカルの remote-tracking branch を使う旨を明示する。
4. **現在のブランチ取得**: `git branch --show-current`
5. **base と head が同じならエラー停止**: 「ベースと比較対象が同一です」と報告して停止。
6. **マージベースの取得**: `git merge-base <BASE_BRANCH> HEAD`
7. **差分の取得**:
   - 変更ファイル一覧: `git diff --name-status <merge-base>...HEAD`
   - 統計: `git diff --stat <merge-base>...HEAD`
   - 詳細diff: `git diff <merge-base>...HEAD`（大きすぎる場合はファイル単位で分割取得）
8. **コミット履歴**: `git log --oneline <merge-base>..HEAD`
9. **変更の影響範囲（コールグラフ）の把握**:
   - 差分から**変更/追加/削除された公開シンボル**を抽出する: 関数名、クラス名、エクスポートされた識別子、HTTP エンドポイントのパス、CLI フラグ、環境変数名、設定キー名、DB スキーマ要素。
   - 各シンボルについて、`git grep` または `rg` で**呼び出し元・参照箇所**を洗い出す:
     ```bash
     # 例: 関数名 foo の参照を、差分外のファイルから探す
     git grep -n "\bfoo\b" -- ':!path/to/changed/file.py'
     ```
   - 影響範囲リストを作成し、Phase 1 の各観点（特に compatibility、architecture、test-coverage、infrastructure）でこれを参照する。
   - 影響範囲が広すぎて全件確認できない場合は、件数と代表サンプルを示し、ユーザーに優先順位を確認する。
10. **大量変更時の方針宣言**: 変更が大きい場合（目安: 1000行超 or 30ファイル超、ただし**プロジェクトの平均的なPRサイズと比較して大きい場合**も含む）は、その旨を述べ、**ファイル単位で順に読む方針**に切り替えることをユーザーに伝える。しきい値が不適切と感じたらユーザーに上書きを求める。

### Phase 1: 観点ごとのレビュー（第1パス）

PERSPECTIVES に含まれる各観点について、以下のテンプレートで独立した評価を行う。**他の観点に引きずられないよう、観点ごとに視点をリセットして読むこと**。

#### 観点別のチェックリスト

##### security（攻撃者目線）
あなたは外部の攻撃者である。この差分から **どうやってこのシステムを壊す/盗む/騙すか** を考えよ。
- 入力検証の欠如（インジェクション: SQL, OS Command, LDAP, XPath, NoSQL, テンプレート）
- 認証・認可の抜け道（権限チェック忘れ、IDORの可能性、トークン検証の不備）
- 機密情報の漏洩（ハードコードされた秘密、ログ・エラーメッセージへの露出、デバッグ情報）
- セキュアでないデフォルト（暗号アルゴリズム、TLS設定、CORS、Cookie属性）
- 競合状態・TOCTOU
- 信頼境界の越境（外部入力が検証なく内部処理へ）
- 依存ライブラリの既知CVE（バージョンが疑わしいものは指摘）
- SSRF / XXE / Open Redirect / パストラバーサル
- レート制限・ブルートフォース耐性
- 監査ログの欠如（誰が何をしたか追跡可能か）

##### maintainability（保守性）
あなたは**6か月後の自分、または新しく入った担当者**である。バグ修正や機能追加でこのコードに戻ってくる立場で評価せよ。
- 変更の影響範囲が局所化されているか（モジュール境界の妥当性）
- 重複コードの発生、DRY違反
- マジックナンバー・マジック文字列
- 命名の一貫性（既存コードベースの慣習との整合）
- 設定とロジックの分離
- 「とりあえず動く」コードや TODO/FIXME の放置
- 廃止予定APIの使用

##### supply-chain-attack（バックドア・悪意あるコード混入の検出）
あなたは**敵対的なコード混入を疑う防御セキュリティ専門家**である。security 観点が「うっかりミス由来の脆弱性」を扱うのに対し、本観点は**意図的な悪意ある混入**を別軸で評価する。差分レビューでは**新しく追加されたコードに潜むバックドアの瞬間を捉える**ことが本質的に重要。

**重要な前提**:
- 本観点は SAST / SCA / シークレットスキャン等の専用ツールの**代替ではなく補完**である。これらの並走を前提とする。
- 検出は「**疑い**」として記録し、確信できない場合も人間のエスカレーションを推奨する（後述のレポートテンプレート参照）。
- 既存コードの慣習を無視した「**いつもと違うコード**」が最も疑わしい。差分レビューはこの違和感を捉えるのに適している。

###### Part 1: 基本検出（ハードコードされた悪意の兆候）

- **ハードコードされたバックドア認証**:
  - `if user == "admin_secret"`、`if password == "<固定値>"`、`if token == "skip_auth_xyz"` 等の特定値での分岐
  - コメントに `# for debug`, `# temporary` などが付いた認証スキップ
  - 環境変数で条件分岐するが、その変数が `.env.example` などにも記載されていない（隠し設定）
- **隠しエンドポイント / 隠しコマンド**:
  - `/admin/_internal_`、`/debug/`、`/.well-known/` 配下に追加された不審なパス
  - 認証ミドルウェアが**意図的にスキップ**されているルート
  - CLI で `--secret-mode`、`__hidden__` のような隠しフラグ
- **動的コード実行への信頼できない入力**:
  - `eval`, `exec`, `Function()`, `new Function`, `os.system`, `subprocess(shell=True)`, `child_process.exec`
  - これらに**外部入力やリクエストパラメータ**が渡される箇所
  - 文字列連結による動的SQL（プレースホルダー未使用）
- **リバースシェル / アウトバウンド接続パターン**:
  - `socket.connect`, `nc -e`, `bash -i >& /dev/tcp/`, `curl ... | sh`, `wget ... | bash`
  - 外部 IP / ドメインへの接続（**コード内のハードコード IP** は特に疑う）
  - WebSocket・SSE などで予期しない外部接続
- **シェル/コマンドインジェクション経路**:
  - ユーザー入力をシェルに渡す箇所
  - `os.system`, `subprocess.run` で shell=True
  - ファイル名・パスへの未検証入力（パストラバーサル）
- **暗号の意図的弱化**:
  - 強い暗号から弱い暗号への差し替え（AES → DES、SHA-256 → MD5 等）
  - 乱数源の差し替え（`secrets` → `random`、`crypto.randomBytes` → `Math.random`）
  - 鍵長の意図的縮小

###### Part 2: 情報漏洩経路の検出

- **外部送信の追加**:
  - HTTP リクエスト送信先が**新しく追加された URL**（特にハードコード IP、見慣れないドメイン）
  - DNS リクエスト（`dns.lookup` で外部ドメイン）
  - WebSocket / gRPC で予期しないエンドポイント
- **ログ・エラーへの機密混入**:
  - パスワード、トークン、秘密鍵、内部 ID、PII（個人情報）がログ出力されている
  - スタックトレースに変数値が含まれ、機密が露出
  - エラーレスポンスに過剰情報（DB エラーメッセージそのまま返す等）
- **デバッグ情報の本番混入**:
  - `console.log` / `print` / `pp` の残骸
  - デバッグ用ルート（`/debug`, `/__debug__/`）
  - スタックトレースのレスポンスへの含有
- **データ持ち出しの兆候**:
  - 全ユーザーデータを取得するエンドポイント・関数の追加
  - 大量データを base64 等でエンコードして外部送信
  - 通常 GET で取得できない情報を返す追加エンドポイント

###### Part 3: 高度なパターン（巧妙な混入の検出）

- **隠れた転送処理**:
  - 中間関数で副作用として外部送信（一見ログ出力のように見える関数が実は外部に転送）
  - エラーハンドラ内に追加された外部通信（通常時は気づきにくい）
  - 定期実行（タイマー、cron）に追加された不審な処理
- **不思議な処理連鎖**:
  - 関連性が低い処理が**1コミットで一緒に追加**されている（例: ユーザー認証修正に DNS lookup が混ざる）
  - 単純な変更のはずなのに**広範囲のファイルが touch**されている
  - **インポート文だけが追加**されたファイル（依存関係の操作）
- **隣接コミットとのギャップ**:
  - 直前のコミットと**スタイルが大きく異なる**（命名規約、コメント形式、ブレーススタイル）
  - 1人のコミッターが**急に他人の領域を変更**している
  - **コミットメッセージと実装内容の乖離**（"typo fix" コミットで実装が大きく変わっている）
- **コミットシグネチャ検証**:
  - `git log --show-signature <merge-base>..HEAD` で署名状況を確認
  - 署名されていないコミットや、署名者が普段と違うコミット
  - GPG 署名の検証失敗（`gpg.badsignature`）
- **サプライチェーンシグネチャ検証**:
  - `package.json`, `requirements.txt`, `Cargo.toml`, `go.mod` での依存追加・更新
  - **新規追加された依存**: パッケージ名の妥当性（typosquatting: `lodash` → `lodahs`、`requests` → `requestz`）
  - **バージョン固定の解除**: `1.2.3` → `^1.2.3` または `*` への変更
  - **メンテナの変更**（リポジトリの所有者が変わった、メンテナンスが急に活発化した依存）
  - **ロックファイルの矛盾**（package.json は変わっていないのに lockfile だけ書き換わっている）
  - **postinstall / preinstall / postpublish** スクリプトの追加

###### Part 4: 評価レポートテンプレート（人間へのエスカレーション）

検出した疑いを**1件1件レポート化**する。テンプレート:

```
### 🚨 疑い: [簡潔なタイトル]

**ファイル**: path/to/file.py:42
**追加コミット**: <SHA> by <author> (<date>)
**確信度**: 高 / 中 / 低
**カテゴリ**: ハードコードバックドア / 認証バイパス / 動的コード実行 / リバースシェル風 / 情報漏洩経路 / 隠れた転送 / 不思議な処理連鎖 / 隣接コミットギャップ / コミット署名異常 / 依存追加 / postinstall / その他

#### 観測された事実
- 具体的なコード片や挙動を引用する
- 検出のトリガーになったパターンを明示する

#### なぜ怪しいのか
- 通常の実装との違い
- 攻撃成立の経路（具体的に）

#### 代替解釈（誤検知の可能性）
- 同じパターンが正当である場合の説明
- 何を確認すれば誤検知と判断できるか

#### 推奨アクション
- マージ阻止 / 追加レビュー / 関係者確認 / ペアレビュー / SAST 実行 / 依存パッケージの監査
- エスカレーション先（セキュリティチーム、メンテナ等）
```

複数の疑いがある場合は**確信度の高い順**にまとめる。**1件もなくても、観点としてスキャンしたことを必ず明示**する（「supply-chain-attack: スキャン完了、疑わしい変更は検出されず」）。

###### 動作上の補足

- **誤検知を恐れて沈黙しない**: 疑いは疑いとして報告し、判断は人間に委ねる。「これは多分問題ない」で省略しないこと。
- **過剰判定も避ける**: あらゆる `eval` を Critical にしない。**コンテキスト**（外部入力の有無、認証の有無、サンドボックスの有無）を見て判定。
- **「疑い」と「断定」を区別**: 確信できない場合は「疑い」と明示し、断定的な指摘は確実な証拠がある場合のみ。
- **専用ツールの並走を推奨**: 出力末尾に「Semgrep、CodeQL、gitleaks、Snyk 等の専用ツールとの併用を推奨」と明記する。
- **コミット署名の検証は補助的**: 署名がないこと自体は問題ではない（多くのプロジェクトで署名は任意）。**普段は署名されているのに今回だけ署名なし**といった**変化**が疑わしい。

##### readability（可読性）
あなたは**入社初日の新メンバー**である。コミット履歴も設計ドキュメントも見ずに、コードと最小限のコメントだけで意図を読み取れるか評価せよ。
- 関数・変数名が意図を表しているか
- 関数の長さ・引数の多さ・ネストの深さ
- 早期returnで複雑性を下げているか
- ブール引数・タプル戻り値などの不透明な表現
- 1ファイルに詰め込みすぎていないか
- 自然言語コメントが「何を」ではなく「なぜ」を説明しているか

##### test-coverage（テスト網羅性）
あなたは**QAエンジニア**である。実装が「テストされた挙動」と「されていない挙動」を分け、後者がリリースされるリスクを評価せよ。
- 新規・変更コードに対応するテストが存在するか
- ハッピーパスだけでなく、エッジケース・異常系・境界値が含まれているか
- 公開APIの全分岐がカバーされているか
- 削除されたコードに対応するテストも適切に削除/更新されているか
- **PBT 採用箇所の読み替え**: PBT で検証されている関数では「分岐網羅」ではなく「性質網羅」で評価する。すなわち、その関数が満たすべき不変条件・代数法則・ラウンドトリップ性のうち、どれが性質として表明されていないかを問う。例示テストでカバーすべき境界値（過去の回帰、特定の業務ルール閾値）が、PBT に吸収されてしまって**例示として残っていない**ケースは指摘対象。

##### test-quality（テスト品質）
あなたは**懐疑的なテストレビュアー**である。テストが「通った」ことではなく「何を保証しているか」を疑い、信頼に値するか評価せよ。網羅性とは別軸で、テスト自体が**信頼できるか**を見る。
- アサーションが本質を検証しているか(実装の偶然の挙動を固定化していないか)
- 過剰なモック(モックがモックを返すような構造)
- テスト名が「何を保証するか」を表現しているか
- Flaky になりうる要素(時刻、乱数、外部I/O、順序依存)
- 1テストで多くを検証しすぎていないか
- セットアップが複雑すぎてテスト意図が読めなくなっていないか
- **PBT 採用箇所の読み替え**: PBT は意図的に乱数を使うため「乱数 = Flaky」とは判定しない。代わりに次を見る。(a) 失敗時の**シード値が再現可能な形で出力される**か、(b) 生成器が外部状態(時刻、グローバル変数、ファイルシステム)に依存していないか、(c) shrinking が動作する純粋関数になっているか。これらが満たされていれば「乱数を使う PBT」は健全。逆に、PBT を装って Date.now() や Math.random() を生成器外で使っていたら Flaky 認定。

##### test-strategy（テスト戦略: 例示ベース vs プロパティベース）
あなたは**テスト戦略の設計者**である。「どのテストを例示ベースで書き、どのテストをプロパティベースで書くべきか」の選択の妥当性を、独立した思考プロセスで評価せよ。このプロジェクトはプロパティベーステスト（PBT）を採用している。両者は対立ではなく補完関係である前提で見る。

**プロパティベースが向いている対象（PBTで書くべきもの）**:
- 代数的法則を持つ関数（結合律・可換律・分配律・冪等性・恒等性）
- ラウンドトリップ性（encode/decode、serialize/deserialize、parse/print、compress/decompress）
- 不変条件を持つデータ構造の操作（ソート後は順序付き、追加後にサイズが+1、削除後に要素が含まれない 等）
- メタモルフィック関係（入力をXに変換しても結果がY倍になる等の関係性）
- 既存実装との等価性検証（リファクタリング前後、最適化版 vs 単純版、複数実装の同値性）
- 状態遷移系（モデルベースPBT: 操作列に対する事後条件）
- 純粋関数で入力空間が広いもの

**例示ベースが向いている対象（PBTにすべきでないもの）**:
- 具体的な値の対応関係を文書化する目的のテスト（仕様の例示・回帰防止）
- バグ修正の再現ケース（過去に壊れた具体的な入力を固定化する）
- ビジネスルールの境界値（特定の閾値・カットオフ・税率変更など）
- 外部システムとの統合（決定論的でない、生成器が現実的でない）
- UI/E2Eシナリオ（人間が読んでフローを理解する目的）
- パフォーマンス特性のスナップショット

**指摘すべき具体的なアンチパターン**:
- 例示ベースで書かれているが、本来PBTで法則を捉えるべき箇所（例: ソート関数を3ケースだけでテスト → 順序性・要素保存・冪等性の性質をPBT化すべき）
- PBTで書かれているが、生成器の制約が強すぎて事実上1〜2ケースしか生成されていない（PBTを装った例示テストになっている）
- プロパティが**実装の写像**になっている（実装と同じロジックを別の書き方で検証 → トートロジー）
- 「すべての入力で例外を投げないこと」など、**性質として弱すぎる**プロパティ
- 反例最小化（shrinking）が機能しない生成器（共有状態、副作用、リファレンス等価性への依存）
- 失敗時の再現性が担保されていないか（シード値の固定・出力・CIでの再現手順）
- 試行回数が少なすぎる/多すぎる（CI時間とのバランス）
- 生成器の分布が偏っていて重要な入力空間（境界値・空・極大・Unicode・NaN等）に到達しない
- 同一の性質を例示テストとPBTで二重に書いていないか（保守コストの増大）
- 回帰ケースをPBTのプロパティに混ぜている（個別の回帰は例示テストとして残すべき）
- プロパティ名が「何の性質か」を表現していない（例: `prop_test1` → `prop_sort_preserves_multiset`）
- 不変条件・事前条件・事後条件の区別が曖昧（特に状態遷移系のテストで）

**判断軸の質問**:
- このコードに「常に成り立つ性質」があるか? あるならPBTの第一候補。
- このテストは「仕様の例示」か「性質の保証」か?
- 入力空間が広いのに例示が3〜5ケースで止まっていないか?
- PBTの失敗が起きたら開発者は原因にたどり着けるか?（shrinking、再現手順）

##### performance（パフォーマンス）
あなたは**本番データ規模を知っているパフォーマンスエンジニア**である。開発時には問題にならないが本番では破綻するパターンを探せ。
- N+1クエリ・ループ内I/O
- 不要な再計算、メモ化の欠如
- O(n²) 以上の見落とし
- 大きなオブジェクトの不要なコピー・全件メモリロード
- インデックスを使わないクエリ・全件スキャン
- 同期処理で十分なところでの不要な非同期化、またはその逆
- キャッシュの妥当性（無効化戦略含む）

##### architecture（アーキテクチャ/設計）
あなたは**プロジェクト全体の構造を把握しているテックリード**である。この変更が長期的にアーキテクチャの一貫性を保つか、徐々に崩していくかを評価せよ。
- 関心の分離（プレゼンテーション/ドメイン/インフラの混在）
- 依存方向（内側→外側になっていないか、循環依存）
- 単一責任原則の違反
- レイヤー違反（コントローラからDBアクセス直叩き等）
- 既存パターンとの一貫性
- 抽象化のレベルが揃っているか
- インターフェイス分離（不要な巨大インターフェイスへの依存）

##### ddd-tactical（DDD戦術的設計とユビキタス言語）
あなたは**ドメインモデルの番人**である。ドメインの意図がコードに正しく現れているか、ユビキタス言語が一貫しているか、戦術的パターンが目的通りに使われているかを評価せよ。本観点は**プロジェクトが DDD を採用している前提**で評価する。コードベースに集約・エンティティ・値オブジェクト等の語彙や構造が見られない場合、または採用が確認できない場合は、その旨を明示して評価をスキップしてよい。

**集約（Aggregate）**:
- 集約境界が**不変条件**を保護する単位として妥当か。境界をまたいだ強整合性の要求がないか（要なら境界の見直し）。
- **集約ルート**を経由しない子エンティティの直接更新がないか。
- 集約サイズが過大でないか（読み込み・ロックコストの肥大化）、または過小で不変条件を守れていないか。
- 集約間の参照は**ID参照のみ**になっているか（オブジェクト参照で結合していないか）。
- 1トランザクション = 1集約 の原則が守られているか（data-integrity 観点と相互参照）。
- 集約内の状態変更が**集約ルートの振る舞い**として表現されているか（外部からのフィールド直接書き換えになっていないか）。

**エンティティ（Entity） vs 値オブジェクト（Value Object）の使い分け**:
- 同一性で識別すべきものがエンティティになっているか（ID を持つか）。
- 属性の集まりで同一性が決まるものが値オブジェクトになっているか。
- 値オブジェクトが**不変（イミュータブル）**として実装されているか。setter や mutable な状態がないか。
- 値オブジェクトに**振る舞い**が定義されているか（Money の加算、DateRange の重なり判定等）。データ構造として箱になっていないか（Anemic Domain Model の兆候）。
- 「Primitive Obsession」: メールアドレス、金額、日付範囲などが文字列や数値のままになっていないか。

**ドメインサービス vs アプリケーションサービス**:
- **ドメインサービス**: ドメインの概念だが特定のエンティティ・値オブジェクトに属さないロジック（例: 振込処理）が、適切にドメイン層に置かれているか。
- **アプリケーションサービス**: ユースケースの調整役（トランザクション境界、外部システム呼び出しの順序）が、ドメインロジックを含んでいないか。
- 両者の混在: アプリケーションサービスにビジネスルールが書かれている、ドメインサービスがインフラに依存している等。

**リポジトリ（Repository）**:
- リポジトリは**集約単位**で定義されているか（テーブル単位になっていないか）。
- リポジトリのインターフェイスがドメイン層に、実装がインフラ層にあるか（依存方向の遵守）。
- リポジトリにビジネスロジック（複雑な条件分岐、計算）が混入していないか。
- リポジトリのメソッド名がコレクション操作（add, remove, find）になっているか、CRUD（insert, update, delete）になっていないか。
- N+1 を生む取得メソッドの設計（performance 観点と相互参照）。

**ドメインイベント（Domain Event）**:
- ドメイン側で発生する重要な出来事がイベントとして表現されているか。
- イベント名が**過去形**（OrderPlaced、PaymentCompleted）になっているか。
- イベントが**不変**で、発生時の事実を表しているか。
- イベント発行と DB 更新の順序（outbox パターン、data-integrity と相互参照）。
- イベントハンドラがイベントの責務を逸脱していないか。

**ファクトリ（Factory）**:
- 複雑な生成ロジックがファクトリに集約されているか、各所に散らばっていないか。
- ファクトリが不変条件を満たしたインスタンスを返すか（不正な状態のオブジェクトを作らないか）。

**仕様オブジェクト（Specification）**:
- 複雑なドメインルールが Specification として再利用可能な形に切り出されているか（必要なら）。
- 仕様の組み合わせ（AND/OR/NOT）が表現できる構造か。

**ユビキタス言語の一貫性**:
- ドメインエキスパートが使う用語と、コード（クラス名、メソッド名、変数名）の用語が一致しているか。
- **同一概念に複数の名前**が使われていないか（User と Customer と Account が同じものを指している等）。
- **異なる概念に同じ名前**が使われていないか（Order が「注文」と「並び順」の両方）。
- 翻訳の不統一（英語クラス名と日本語コメントで用語がずれている）。
- 専門用語の説明があるか（ドキュメント、ADR、コードコメント）。
- データベースのカラム名・API のフィールド名もユビキタス言語に従っているか（境界での裏切りがないか）。

**よくあるアンチパターン**:
- **Anemic Domain Model**: エンティティが getter/setter だけで振る舞いが Service にすべて吸われている。
- **God Aggregate**: 1つの集約が巨大化し、システムの大半を含んでいる。
- **トランザクションスクリプト化**: ドメイン層が薄く、アプリケーションサービスに手続き的なコードが集中している。
- **インフラ汚染**: ドメイン層が ORM の基底クラス・フレームワーク固有の型に依存している。
- **境界の漏れ**: 別の境界づけられたコンテキストのエンティティを直接 import している（ACL なしで結合）。

**動作上の補足**:
- DDD 採用が不明確な場合、変更されたファイルに `Aggregate` / `Entity` / `ValueObject` / `Repository` / `DomainEvent` / `domain/` ディレクトリ等の手がかりがあるか確認する。なければ「DDD 採用の証跡が見当たらないため評価をスキップ」と明示する。
- DDD 用語を**コードに無理矢理当てはめない**こと。「これは集約と呼べる」と推測で指摘するのではなく、コードベースが意図的に DDD を採用しているかを判断材料に。

##### documentation（ドキュメント/コメント）
あなたは**この機能を初めて使う外部開発者**である。コードだけ読んで使えるか、ドキュメント無しで困らないかを評価せよ。
- 公開API・型のドキュメンテーションコメント
- README・CHANGELOG の更新漏れ
- アーキテクチャ図・ADR の更新要否
- コメントとコードの乖離
- 削除されるべき古いコメントの残存

##### error-handling（エラーハンドリング）
あなたは**深夜にこのコードのアラートで起こされた SRE** である。エラーが起きたとき、何が壊れて、どう復旧するかが追えるかを評価せよ。
- 例外の握りつぶし（catch して何もしない、ログだけで継続）
- 過剰に広い catch（全例外を一括で握る）
- リソースリーク（ファイル・コネクション・ロックの解放漏れ）
- リトライ戦略（指数バックオフ、最大回数、冪等性）
- ユーザー向けエラーメッセージと内部ログの分離
- エラー時の状態の一貫性（部分的な状態変更が残らないか）

##### data-integrity（データ整合性・トランザクション・冪等性）
あなたは**「データが壊れたら誰が責任を取るのか」を考える DBA / 信頼性エンジニア**である。**実行時にデータが壊れる/不整合になる経路**を探す。バックエンド変更で最も事故になりやすい領域。
- **トランザクション境界**: 複数の更新が atomic か。トランザクション内に外部APIコールが含まれていないか（長時間ロックの原因）。
- **トランザクション伝播**: ネストしたトランザクションの伝播仕様が正しいか（REQUIRES_NEW / NESTED など）。
- **ロック戦略**: 楽観ロック（version列）/悲観ロック（SELECT FOR UPDATE）の選択は妥当か。デッドロック順序の一貫性。
- **Read-Modify-Write 競合**: lost update の可能性（取得→計算→書き戻しの間に他プロセスが更新できる構造）。
- **冪等性**: 同じリクエストを2回実行しても結果が同じか（リトライ・ネットワーク再送に耐えるか）。冪等キーの設計。
- **イベントとDB更新の順序**: outbox / transactional outbox パターン未使用による「DB更新成功・イベント発行失敗」または逆の不整合。
- **At-least-once / At-most-once / Exactly-once** の選択と、受け手側の重複排除設計。
- **部分的失敗の補償**: Saga パターン、補償トランザクション、リカバリ手順。
- **整合性の種類**: 強整合性/結果整合性の使い分け。結果整合性で良いはずの箇所で強整合性を要求していないか（性能劣化の原因）。
- **時刻の一貫性**: クライアント時刻 vs サーバー時刻、タイムゾーン、夏時間、クロックドリフトの影響。
- **マイグレーションの安全性**: スキーマ変更とコード変更の順序（ローリングアップデート可否）、ロールバック時の整合性。
- **データの取りこぼし**: バッチ処理の境界（last_processed_id の更新タイミング、worker クラッシュ時の再処理範囲）。
- **集計値とソースの整合**: キャッシュ・マテリアライズドビュー・サマリーテーブルの更新漏れ。

##### compatibility（後方互換性/破壊的変更）
あなたは**既存利用者**である。このリリースで自分のシステムやワークフローが壊れないかを評価せよ。ブランチ間diff だからこそ価値が高い観点。
- 公開API（関数シグネチャ、HTTPエンドポイント、CLIフラグ）の破壊的変更
- DBスキーマのマイグレーション（後方互換性、ロールバック可否）
- 設定ファイル形式の変更（既存設定での起動可否）
- イベント・メッセージスキーマの変更
- デフォルト値の変更による既存利用者への影響
- バージョニング・非推奨化の手順を踏んでいるか。具体的には**(a) SemVer に従ったバージョン番号の更新**、**(b) 非推奨期間（deprecation period）を設けた段階的廃止**、**(c) 移行ガイド（migration guide）の提供**、**(d) CHANGELOG での破壊的変更の明示**、の4点を確認する。破壊的変更を非破壊的に見せていないか（例: メジャー更新が必要な変更をマイナーで出している）。

##### dependencies（依存関係）
あなたは**サプライチェーン攻撃を警戒するセキュリティエンジニア**である。同時に、5年後にこの依存をメンテし続ける開発者の立場も持ち合わせよ。
- 新規追加された依存の必要性（標準ライブラリで足りないか）
- ライセンスの互換性（GPL系の混入など）
- メンテ状況（過去1年更新がない、Star数極小、単一メンテナ）
- バージョン固定 vs レンジ指定の妥当性
- ロックファイルの更新漏れ
- サプライチェーン上のリスク（typosquatting、新規パッケージ、postinstall スクリプトの有無、SBOM、署名付きパッケージの検証）

##### code-provenance（AI生成コード・コピペコード起源のリスク）
あなたは**懐疑的なシニアレビュアー**である。コードが「もっともらしく見える」だけで実際は機能しない、または別の文脈から流用されただけで現在の文脈に合っていない可能性を疑え。このコマンドは Claude Code から呼ばれる前提のため、**変更の一部または全部が AI 生成である可能性**を常に想定する。AI 生成コード特有のリスクと、人間によるコピペ起源のリスクを併せて評価する。
- **幻覚 API**: 実在しないライブラリ関数、引数順序の誤り、戻り値型の取り違え。特に標準ライブラリの似た名前の関数(`strftime`/`strptime`、`map`/`flatMap` 等)、minor バージョン差で挙動が違う API。**疑わしいシンボルは依存関係のドキュメントで実在を確認**。
- **過剰実装 / YAGNI 違反**: 要求されていない機能の追加、使われない引数・オプション・抽象化レイヤー、想定されていないエッジケースへの過剰な防御コード。
- **コピペコード**: 既存コードベース内の他の関数とほぼ同一のロジックが新規追加されていないか。`grep` / `rg` で類似パターンを探す価値あり。
- **転用ミス**: 別ファイル・別レイヤーから持ってきたコードが、その文脈の前提を引きずっていないか(例: フロントエンドのバリデーションロジックをバックエンドにそのまま流用、テストコード由来のモック設定が本番コードに混入)。
- **テストの自作自演**: 実装とテストを同じ AI が同時生成した場合、テストが「実装が偶然そう動いていること」を固定化しているだけのケース。`test-quality` 観点と相互参照し、**実装と独立した仕様の表現になっているか**を厳しく見る。
- **コメントとコードの乖離**: AI は説明的コメントを付けがちだが、後の編集でコードだけ変わってコメントが残るケース。誤誘導コメントは無コメントより有害。
- **エラーメッセージの不自然さ**: 「Something went wrong」「An error occurred」など汎用すぎるメッセージ、または逆に丁寧すぎてユーザー向けでない技術用語の混入。
- **不要なログ / デバッグコードの残骸**: `console.log`、`print(...)`、`pp(...)`、コメントアウトされたコード片。
- **ライセンス汚染リスク**: 学習データ由来の特徴的なコード塊(コメントの言い回し、変数名の癖、特定 OSS の構造をそのまま再現)。完全な検出は困難だが、**疑わしさを記録**して人間判断を促す。
- **コミット粒度の不自然さ**: 1コミットで広範囲に touch している(AI による一括生成の典型)、コミットメッセージが生成物の体裁だけ整っていて意図が読めない。
- **既存パターンとの不整合**: プロジェクトの既存コードでは A スタイルなのに、追加部分だけ B スタイルになっている(命名規約、エラーハンドリング方針、ログ方針の差異)。

##### infrastructure（インフラ・環境設定・CI/CD）
あなたは**本番デプロイを実行する運用エンジニア**である。「コードは動くが本番で起動できない」「ローカルで動くが CI で落ちる」リスクを評価せよ。本プロジェクトは **Docker compose + VSCode devcontainer 前提**である。コードだけでなく**実行環境の変更**を独立した観点で精査せよ。デプロイ事故の主因はここに集中する。
- **環境変数の変更**:
  - 新規追加された環境変数が `.env.example` / `compose.yml` / `devcontainer.json` / CI設定すべてに反映されているか
  - デフォルト値の変更で既存デプロイに影響しないか
  - 削除された環境変数の参照箇所が残っていないか
- **シークレット管理**:
  - 平文の API キー・トークン・パスワード・秘密鍵が変更ファイルに含まれていないか（`.env` のコミット、Dockerfile への `ENV` 直書き）
  - シークレット参照の方法が既存パターンと一貫しているか
- **Dockerfile / compose.yml の変更**:
  - ベースイメージのバージョン固定（`:latest` の使用は要警告）と更新の妥当性
  - レイヤー順序の最適化（依存インストールとソースコピーの順序、キャッシュ効率）
  - マルチステージビルドの活用（最終イメージに不要なビルド成果物が含まれていないか）
  - 不要な root 権限・特権モード
  - ボリュームマウント・ポート公開の妥当性
  - ヘルスチェックの定義
- **devcontainer.json の変更**:
  - postCreateCommand / postStartCommand の冪等性
  - 拡張機能・設定の追加がチーム合意に沿っているか
  - features の追加（追加された feature が必要か、セキュリティ上問題ないか）
- **CI/CD パイプライン**（`.github/workflows/`, `.gitlab-ci.yml` など）:
  - シークレットの取り扱い（環境変数経由、マスキング、artifacts への漏洩）
  - 権限スコープの最小化（GITHUB_TOKEN の permissions、OIDC 利用）
  - キャッシュキーの妥当性
  - third-party action の pin（tag ではなく commit SHA で固定されているか）
  - matrix の網羅性とコスト
  - 失敗時の通知・ロールバック手順
- **マイグレーションスクリプト**:
  - 適用順序が明確か（タイムスタンプ・連番）
  - 冪等に実行できるか（再実行で壊れないか）
  - ロールバックスクリプトの有無と動作確認
  - 大規模テーブルへの破壊的変更（ロック時間、ダウンタイム）への配慮
- **本番デプロイで起動失敗するリスク**:
  - 新規必須環境変数の未設定で起動エラーになる可能性
  - ボリューム/ネットワーク設定の互換性
  - ヘルスチェックパスの変更

##### i18n-a11y（国際化/アクセシビリティ）
あなたは**英語以外を母語とするユーザー、またはスクリーンリーダーで操作するユーザー**である。このコードが言語・身体的状況の違いを超えて使えるかを評価せよ。
- ハードコードされた文字列（翻訳リソースに入れるべきもの）
- 日付・通貨・数値フォーマットのロケール依存
- 文字列連結による翻訳不能な構造
- alt属性、ARIA属性、ラベル、コントラスト、キーボード操作可否
- フォーカス管理・スクリーンリーダー対応

##### observability（ログ/可観測性）
あなたは**本番障害の事後分析（ポストモーテム）を書く担当者**である。障害が起きたとき、ログ・メトリクス・トレースから原因にたどり着けるかを評価せよ。
- 重要処理の入出力・所要時間のログ
- 構造化ログか（grep前提の文字列でないか）
- ログレベルの妥当性（DEBUG/INFO/WARN/ERRORの使い分け）
- 相関ID・トレースIDの伝播
- 機密情報のログ混入（Phase 1の security と相互参照）
- メトリクス・アラートの追加要否。判断軸として**ゴールデンシグナル（レイテンシ、トラフィック、エラー、サチュレーション）**または **RED（Rate, Errors, Duration）/USE（Utilization, Saturation, Errors）**のうち、どれが該当する変更かを意識する。
- **SLO/SLI への影響**: 既存の SLO 指標を変える変更か（エラーレートの定義変更、レイテンシ計測点の移動）。SLO に影響する変更なら、関係者への通知や定義更新の要否を指摘。
- **アラート設計**: 新規アラートを追加する場合、しきい値の根拠、誤検知率、runbook の有無を見る。「とりあえずアラートを追加」は将来のアラート疲労の原因。

#### 重大度の判断基準（観点横断）

重大度の判定は **AI ごとにブレやすい**ため、以下を参照基準とせよ。「危険そう」「悪い気がする」といった印象論ではなく、各カテゴリの**具体経路**を説明できる場合のみ高重大度を付ける。

| 重大度 | 一般的な定義 | マージ判断 |
|---|---|---|
| Critical | 本番で**確実に深刻な被害**が出る、または**攻撃成立の具体経路**が説明できる | マージ前に必ず修正 |
| High | 高確率で問題化する、または被害が大きいが攻撃/障害の前提条件がある | 強く修正推奨、マージ前対応が望ましい |
| Medium | 将来の保守・拡張で問題化する、現状は動作するが不健全 | 同 PR または近い時期に対応 |
| Low / Info | スタイル・好み・将来の改善提案 | 別 PR で良い |

#### 観点別の Critical / High 例（参照基準）

| 観点 | Critical の例 | High の例 |
|---|---|---|
| security | 認証バイパス、リモートコード実行、機密データの平文露出、未パッチの SQL/コマンドインジェクション | CSRF トークン欠如、ハッシュ化されていないパスワード保存、過剰な権限付与、ログへの個人情報出力 |
| supply-chain-attack | 確信度高のバックドア・認証バイパス・リバースシェル風処理・既知の悪意ある依存パッケージの追加 | 確信度中の不審な動的コード実行、不可解な処理連鎖、postinstall スクリプト追加、コミット署名異常 |
| data-integrity | トランザクション欠如により金銭/在庫が壊れる、外部APIをトランザクション内に含めて長時間ロック、冪等性なしの課金処理 | lost update が起きる Read-Modify-Write、強整合性が必要な箇所で結果整合性、マイグレーションの非冪等性 |
| error-handling | catch して握りつぶした結果データが壊れる、リソースリークでサービス停止 | リトライ無限ループ、ユーザー向けに内部スタックトレース露出、エラー時の部分的状態変更が残る |
| compatibility | 公開 API の破壊的変更で既存クライアントが起動失敗、DB マイグレーションが後方非互換 | デフォルト値の変更で挙動が静かに変わる、非推奨化手順を踏まない削除 |
| performance | O(n²) や N+1 が本番データ規模で確実に SLA 超過、無限ループ可能性 | 大規模ループ内 I/O、インデックスを使わない頻出クエリ、メモリ全件ロード |
| infrastructure | シークレットの平文コミット、`:latest` ベースイメージで再現不能、CI で `GITHUB_TOKEN` の write 権限を過剰付与 | 環境変数の `.env.example` への反映漏れ、third-party action の tag 参照（SHA pin なし） |
| dependencies | 既知の CVE が High/Critical の依存追加、ライセンス非互換（GPL の混入） | 単一メンテナで過去1年更新なし、postinstall スクリプトを持つ新規パッケージ |
| code-provenance | 幻覚 API による起動時エラー、実装と同 AI 生成のテストで「動作している風」に見えるが本質を検証していない | 過剰な抽象化レイヤ、既存パターン無視の独自実装 |
| architecture | 循環依存でビルド/起動不能、レイヤー違反でセキュリティ境界が壊れる | 単一責任原則の重度違反、依存方向の逆転 |
| ddd-tactical | 集約境界をまたいだ不変条件違反でデータ整合性破壊、別コンテキストのエンティティを ACL なしで直接結合 | Anemic Domain Model（ドメイン層がデータ箱化）、リポジトリへのビジネスロジック混入、ユビキタス言語の重大な不一致（同一概念に複数名） |
| test-coverage | クリティカルパスにテストが全くない | 異常系・境界値のテスト欠如 |
| test-quality | テストが何も検証していない（アサーション無し、または常に成立） | 過剰モックで実質的に何も検証していない |
| test-strategy | PBT を装って実質1ケースしか生成されていない、プロパティが実装の写像（トートロジー） | 例示テストで書くべき回帰ケースが PBT に吸収されて消えた |
| maintainability | 同 PR 内で広範囲のコード重複、6か月後に誰も触れなくなる構造 | 設定とロジックの混在、マジックナンバー多数 |
| readability | （Critical はまれ。可読性単体で本番影響は出にくい） | 関数長・ネスト深さが極端、ブール引数の濫用で誤用しやすい |
| documentation | （Critical はまれ） | 公開 API のドキュメント欠如、README の起動手順が動かない |
| i18n-a11y | アクセシビリティ法令違反相当（公的サービスで WCAG AA 必須など） | ハードコード文字列の多数追加、スクリーンリーダー対応漏れ |
| observability | （Critical はまれ） | 本番障害の原因特定が不可能になるログ削除、相関 ID の伝播切断 |

**注意**: 上記は**目安**であり、プロジェクトの性質（金融/医療/玩具）によって基準は変動する。判定に迷ったら、低めに付けた上で「もし高めに見るなら理由は X」と注記する。

### Phase 2: セルフレビューパス（第2パス）

**Phase 1 で出した指摘を、別人格で読み直す**。以下を自問し、各項目について**Yes/No と根拠**を明示せよ。

#### 2-A. 見落としチェック（必ず全項目を確認）

1. **全変更ファイルが少なくとも1つの観点で言及されたか?** Phase 0 で取得した変更ファイル一覧を再度突き合わせ、一度も登場しなかったファイル名を列挙する。ゼロでない場合、そのファイルが本当に評価不要だったか説明する。
2. **削除されたコード/関数を評価したか?** diff の `-` 行や `D` ステータスのファイルを見たか。削除は追加と同じくらい影響範囲がある（呼び出し元の壊れ、機能の喪失、テストの空洞化）。
3. **テストファイル自体を観点別に評価したか?** テストコードもコードであり、test-quality だけでなく security（テストフィクスチャに本物の機密が混入）、readability、maintainability、performance（テスト時間の肥大化）の対象。
4. **リネーム/移動の影響範囲を確認したか?** `git diff` の `R` ステータス、文字列参照（クラス名の動的解決、設定ファイル内のパス、ドキュメント内の参照）の追従漏れ。
5. **コード以外の変更を見たか?** マイグレーション、設定ファイル、CI/CD 設定、Dockerfile、devcontainer.json、ドキュメント、テストデータ、サンプルコード、`.gitignore` / `.editorconfig` / `.gitattributes`。
6. **observability の指摘で security と矛盾していないか?** ログ追加が機密情報を含む形になっていないか。

#### 2-B. 誤検知・過剰指摘チェック

7. **コンテキストを読めば問題ない指摘が含まれていないか?** 別の箇所で対処済み、フレームワークが自動で処理する、テスト用コードで本番経路ではない、などの理由で取り下げるべき指摘はないか。
8. **個人的好みと根拠ある指摘を区別したか?** 「私はこの書き方が好き」レベルの主張を Medium 以上で出していないか。
9. **同じ問題を複数観点で重複指摘していないか?** 例: security と data-integrity の両方で同じトランザクション問題を出している場合、より本質に近い観点1つに集約する。

#### 2-C. 重大度の妥当性チェック

10. **Critical を付けたものは攻撃可能性または重大事故の具体経路を説明できるか?** 「危険そう」では不十分。「この入力で、この経路で、この結果が起きる」と書けない Critical は High に格下げする。
11. **High を付けたものに「Critical でない理由」を説明できるか?** 説明できない High は Critical の可能性が高い。
12. **Low/Info のうち、別 PR で扱うべきものはどれか?** マージブロッカーと将来の改善提案を明確に分離する。

#### 2-D. 観点間の整合性チェック

13. **観点間で矛盾する推奨を出していないか?** 例: パフォーマンス最適化が可読性を犠牲にしている、抽象化追加（architecture）が YAGNI（code-provenance）と衝突。衝突がある場合は**どちらを優先すべきか理由付きで決着**させる。
14. **「修正提案」が別の指摘を生まないか?** 例: 「ここを早期 return に」と提案したコードが、別の指摘で挙げた「ネスト深さ」のチェックに引っかかる、など。

#### 2-E. 影響範囲チェック

15. **変更された公開シンボル（関数、クラス、エクスポート、HTTP エンドポイント、CLI フラグ、環境変数）の呼び出し元/参照箇所を確認したか?** Phase 0 step 9 で取得したコールグラフ情報を見直す。compatibility 観点との連動。

#### 2-F. 出力

第2パスの結果は、Phase 1 の各指摘に対する **追加・撤回・修正** として明示せよ:

```
## セルフレビュー結果

### 自問への回答
- 2-A.1: Yes / No - 根拠: ...
- 2-A.2: Yes / No - 根拠: ...
（全15項目）

### 追加された指摘
- ...

### 撤回した指摘
- [Phase 1 の該当指摘] を撤回。理由: ...

### 重大度を変更した指摘
- [指摘] Critical → High。理由: ...

### 観点間の衝突解決
- ...
```

## 進捗表示ルール

実行中はユーザーがハングアップを疑わないよう、**主要ステップごとに必ず1行の進捗ログを出す**こと。以下のタイミングで進捗を出す:

1. **Phase の開始時**: `🔍 Phase 0: 準備（ベースブランチ決定 & 差分の取得）を開始`
2. **Phase 0 の主要ステップ前**: `🔍 ベースブランチを判定中...` / `🔍 マージベースを取得中...` / `🔍 差分を取得中（変更ファイル N 件）...` / `🔍 公開シンボルの呼び出し元を検索中（M 件のシンボル）...`
3. **Phase 0 の完了時**: `✅ Phase 0 完了（変更 N ファイル、M 行）`
4. **Phase 1 の各観点の開始時**: `🔍 [観点名] を評価中...`
5. **Phase 1 の各観点の完了時**: `✅ [観点名] 完了（指摘 N 件）`
6. **Phase 2 の開始時**: `🔍 Phase 2: セルフレビューパスを開始`
7. **Phase 3 の開始時**: `🔍 Phase 3: 横断サマリーを作成中...`
8. **時間がかかるツール実行前**（`git log` で大量履歴取得、`rg` でリポジトリ全体スキャン等）: `🔍 <何をしているか> 実行中...`

### 進捗ログの書式

- **必ず 1 行**で、絵文字（🔍 進行中 / ✅ 完了 / ⚠️ 警告 / ❌ エラー）+ 動詞を含む短文
- 進捗ログは**通常の出力と区別できる行**にする（前後に空行を入れるか、コードブロック外に書く）
- 数値が把握できているなら入れる（「N 件」「M ファイル」「X 観点中の Y 個目」など）
- 進捗ログ自体に分析結果を入れない（結果は所定のフォーマットで別途出力）

### 進捗ログが冗長になる場合

観点が多く出力がノイズになりそうなときは、進捗ログをまとめても良いが、**少なくとも各観点の開始と完了は必ず出す**こと（評価中に黙ったまま長時間経過するのを防ぐ）。

## 動作上の注意

- **推測で進めない**: ベースブランチが特定できない、引数の解釈が曖昧、差分が大きすぎる等の状況では、必ずユーザーに方針を確認してから進めること。
- **観点ごとに視点をリセット**: あるパスの結論を次のパスに持ち込まない。観点ごとの独立性が複数回見直しの価値を生む。
- **コード例は短く具体的に**: 「ここをこう書き直す」と分かる diff スタイルが望ましい。
- **ファイルパスと行番号を必ず添える**: 指摘の再現性のため。
- **不確実なら不確実と書く**: 「攻撃可能かは呼び出し元次第」など、確認すべき前提も明示する。
- **既存コードのスタイル・規約を尊重**: 個人的な好みでの指摘と、根拠ある指摘は区別すること。

## 起動

以上の手順に従い、**Phase 0 から開始**せよ。**最初に進捗ログ「🔍 Phase 0: 準備（ベースブランチ決定 & 差分の取得）を開始」を出してから**、最初の応答は「引数解釈の結果」「ベースブランチの決定根拠」「変更の規模感」「これから実行する観点リスト」の宣言から始めること。
